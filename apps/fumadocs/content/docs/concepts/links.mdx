---
title: Links
description: Declare dependencies between resources
---

**Links** define relationships between resources. They control creation order, enable network connectivity, and express the topology of your architecture.

## Defining Links

```yaml
links:
  - { from: api, to: db, type: depends_on }
  - { from: web, to: api, type: network }
  - { from: worker, to: queue, type: depends_on }
```

## Link Types

| Type | Meaning |
|---|---|
| `depends_on` | Target must be healthy before source starts. Used for databases and caches that app resources depend on. |
| `network` | Source can reach target over the internal network. Used for service-to-service communication. |
| `mounts` | Source mounts target's persistent storage. Used for volume resources. |

## Creation Order

During sync, Otterdeploy uses links to determine safe creation order. Resources are created in dependency order:

1. Databases and caches (no outgoing dependencies)
2. APIs and workers (depend on databases/caches)
3. Web frontends (depend on APIs)

## Link Inheritance

When an environment extends a parent:

- If the child defines `links`, they **replace** all parent links
- If the child doesn't define `links`, parent links are inherited
- Use `extraLinks` to add links without replacing all inherited ones
- Use `removeLinks` to remove specific inherited links by `from`/`to` pair

```yaml
staging:
  extends: production
  # Don't define "links" — inherit from production
  removeLinks:
    - { from: api, to: redis }         # remove this one
  extraLinks:
    - { from: web, to: db, type: network }  # add this one
```

## Automatic Cleanup

Links referencing resources that don't exist (e.g., excluded resources) are automatically removed during inheritance resolution. You don't need to manually clean up links after excluding a resource.
