---
title: YAML Config
description: Write your Otterdeploy config in YAML instead of TypeScript
---

Otterdeploy supports YAML as an alternative to TypeScript for config files. YAML configs are validated through the same Zod schema, so the structure is identical.

## When to Use YAML

- Your team doesn't use TypeScript/JavaScript
- You prefer a language-agnostic config format
- You're generating configs from a CI/CD pipeline or templating tool

## When to Use TypeScript

- You want type checking and autocompletion in your editor
- You need computed values or dynamic logic in your config

## File Naming

Name your file `otterdeploy.yaml` or `otterdeploy.yml`. Otterdeploy checks for YAML files before TypeScript files:

1. `otterdeploy.yaml`
2. `otterdeploy.yml`
3. `otterdeploy.config.ts`
4. `otterdeploy.config.js`
5. `otterdeploy.config.mjs`

## Example

```yaml
project: my-app
organization: org_abc123

environments:
  production:
    resources:
      api:
        kind: api
        source:
          github: my-org/api
          branch: main
          root: /
        build:
          builder: nixpacks
          buildCommand: "npm run build"
          watchPatterns:
            - "src/**"
            - "package.json"
        deploy:
          startCommand: "node dist/index.js"
          preDeployCommand: "npx prisma migrate deploy"
          replicas: 3
          healthcheckPath: /health
          healthcheckTimeout: 30
          restartPolicy: ON_FAILURE
          region: us-west
        domain:
          - api.myapp.com
          - api-v2.myapp.com
        env:
          NODE_ENV: production
          DATABASE_URL: "${db.connectionString}"

      web:
        kind: web
        source:
          github: my-org/web
          branch: main
        build:
          builder: nixpacks
        deploy:
          replicas: 2
        domain: myapp.com
        env:
          API_URL: "https://${api.domain}"

      db:
        kind: database
        image: postgres:16
        storage: 20Gi
        env:
          POSTGRES_DB: myapp

      redis:
        kind: cache
        image: redis:7-alpine

    links:
      - { from: api, to: db, type: depends_on }
      - { from: api, to: redis, type: depends_on }
      - { from: web, to: api, type: network }

  staging:
    extends: production
    exclude:
      - redis
    overrides:
      resources:
        api:
          source:
            branch: develop
          deploy:
            replicas: 1
          domain: api.staging.myapp.com
          env:
            NODE_ENV: staging
        web:
          domain: staging.myapp.com
    extraLinks:
      - { from: web, to: db, type: network }
```

## Gotchas

- Strings containing `${}` references must be quoted: `"${db.connectionString}"`
- Boolean values like `true`/`false` are parsed as booleans, not strings
- Numbers are parsed as numbers — use quotes if you need a string: `"3000"`
